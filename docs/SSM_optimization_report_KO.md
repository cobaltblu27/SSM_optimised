## 3. 무엇을 개선했는가

SSM 파이프라인은 크게 3단계로 구성됨:

```
[1단계] train: 학습 분자들에 대한 random walk 수행 (2,936 분자 × 20회 반복)
[2단계] valid: 테스트 분자들에 대한 random walk 수행 (26,090 분자 × 20회 반복)
[3단계] prediction: 결과 분류 + DiSC 패턴 마이닝
```

각 단계별 소요 시간 변화:

| 단계 | 기존 (Baseline) | 개선 후 (순차) | 개선 후 (병렬) | 주요 개선 내용 |
|------|-----------------|---------------|----------------|---------------|
| train | ~5시간 | ~42분 | ~10분 | 중복 계산 제거 + 병렬 처리 |
| valid | ~5.5시간 | ~144분 | ~93분 | 중복 계산 제거 + 병렬 처리 |
| prediction + DiSC | **~94시간** | **~110분** | **~42분** | 매칭 결과 캐싱 (핵심) |
| **합계** | **104.5시간** | **5시간 16분** | **2시간 25분** | |

> 기존 소요 시간의 **90% (94시간)**를 차지하던 DiSC 패턴 마이닝 단계의 비효율을 해결한 것이 가장 큰 효과.

## 4. 어떤 방법으로 개선했는가

개선은 4가지 방향으로 이루어졌으며, 모두 "같은 계산을 더 효율적으로 수행"하는 것.

### 4-1. 불필요한 반복 제거 (DiSC 매칭 캐싱) — 가장 큰 효과

DiSC 패턴 마이닝은 "어떤 subgraph 조합이 특정 class에서 더 자주 나타나는가"를 찾는 과정임.
이를 위해 fragment(분자의 부분 구조)들의 조합을 만들고, 각 조합이 학습 분자 2,936개 중 어디에 존재하는지를 검사함.

**기존 방식의 문제:**

Fragment가 300개라면, 2개씩 조합하면 약 45,000가지, 3개씩 조합하면 약 450만 가지가 됨.
기존 코드는 **매 조합마다** 2,936개 분자 전부를 대상으로 "이 조합이 이 분자에 존재하는가?"를 처음부터 새로 검사했음.

예시로 설명하면:

```
Fragment 300개, 분자 2,936개일 때:

기존 방식:
  조합 {A, B}를 검사 → 2,936개 분자에 대해 "A와 B가 동시에 있는가?" 매칭 검사
  조합 {A, C}를 검사 → 2,936개 분자에 대해 "A와 C가 동시에 있는가?" 매칭 검사
  조합 {A, D}를 검사 → 2,936개 분자에 대해 "A와 D가 동시에 있는가?" 매칭 검사
  ...
  → 수만~수백만 가지 조합 × 2,936개 분자 = 수십억 번의 매칭 검사
  → 이것이 94시간(4일)이 걸린 원인
```

**개선 방식:**

핵심 관찰: "분자 X에 fragment A와 B가 동시에 존재하는가?"는
"분자 X에 A가 존재하는가?" 그리고(AND) "분자 X에 B가 존재하는가?"와 같음.

따라서 각 fragment 개별 매칭 결과를 처음 1회만 저장해두면, 어떤 조합이든 저장된 결과를 꺼내서 AND만 하면 됨.

```
개선 방식:
  [사전 준비] fragment 300개 × 분자 2,936개 = 약 88만 번만 매칭 검사하여 결과 저장
    - Fragment A: [분자1=O, 분자2=X, 분자3=O, ...]  (O=존재, X=없음)
    - Fragment B: [분자1=O, 분자2=O, 분자3=X, ...]
    - Fragment C: [분자1=X, 분자2=O, 분자3=O, ...]
    - ...

  [조합 검사] 저장된 결과끼리 AND 연산만 수행 (매칭 검사 없이 즉시 완료)
    - 조합 {A, B}: A결과 AND B결과 = [분자1=O, 분자2=X, 분자3=X, ...]
    - 조합 {A, C}: A결과 AND C결과 = [분자1=X, 분자2=X, 분자3=O, ...]
    - ...
    → 수백만 가지 조합이라도 AND 연산은 거의 순간적으로 완료
```

결과적으로 수십억 번의 무거운 매칭 검사가 → 88만 번의 매칭 + 수백만 번의 가벼운 AND 연산으로 바뀜.
이것이 94시간 → 42분으로 단축된 핵심 원리임.

### 4-2. 중복 계산 사전 저장

Random walk에서 반복적으로 동일한 값을 다시 계산하는 부분이 여러 곳 있었음.
비유하자면, "서울에서 부산까지의 거리"를 매번 지도를 펼쳐서 다시 재는 대신, 한 번 재서 메모해두고 이후에는 메모를 보는 것과 같음.

**확률 벡터 캐싱:**

Random walk에서 각 노드(원자)에서 다음 노드로 이동할 확률은 전이 행렬(T)로부터 계산됨.
기존 코드에서는 walker 5명이 각각 동일한 시작점에서 출발할 때, 같은 확률 벡터를 5번 중복 계산했음.
그런데 이 확률 벡터는 "어디에서 시작했는가"와 "몇 번째 step인가"에 의해서만 결정되고, walker가 실제로 어디로 갔는지와는 무관함.
따라서 (시작점, step) 조합별로 1회만 계산하여 저장해두면, 5명의 walker가 모두 저장된 값을 재사용 가능.

```
기존: walker 5명 × 노드 30개 × step 7회 = 1,050번 행렬 계산
개선: 노드 30개 × step 7회 = 210번만 계산하고 저장 → 5배 절감
```

**인접 노드 목록:**

분자 그래프에서 "원자 A와 연결된 이웃 원자 목록"을 매 walk step마다 그래프 구조를 탐색하여 조회했음.
이 목록은 walk 중에 변하지 않으므로, 처음 한 번만 모든 원자의 이웃 목록을 정리해두면 이후에는 즉시 참조 가능.

**Preference DataFrame:**

iteration마다 생성하는 preference 테이블을 기존에는 분자 루프 안에서 분자마다 동일한 것을 반복 생성했음.
이를 루프 밖에서 iteration당 1회만 생성하도록 변경.

```
기존: 2,936 분자 × 19 iteration = 55,784번 동일한 테이블 생성
개선: 19번만 생성
```

### 4-3. 연산 방식 효율화 (벡터화)

행렬의 각 행을 정규화(행의 합이 1이 되도록 나누기)하는 연산이 코드 곳곳에 있었음.
기존에는 Python의 for문으로 행을 한 줄씩 꺼내서 처리했는데, 이를 numpy의 배열 연산으로 전체 행렬을 한 번에 처리하도록 변경.

비유하자면, 학생 1,000명의 시험 점수를 평균내야 할 때:
- 기존: 학생 한 명씩 불러서 점수를 받고, 계산하고, 다음 학생을 부르는 방식
- 개선: 전체 점수표를 한 번에 받아서 일괄 계산하는 방식

계산 결과는 완전히 동일하지만, 컴퓨터가 내부적으로 더 효율적인 방식으로 처리하므로 속도가 향상됨.

### 4-4. 병렬 처리 (`--n_jobs` 옵션 추가)

Random walk는 각 분자에 대해 독립적으로 수행됨. 분자 A의 random walk 결과가 분자 B의 random walk에 영향을 주지 않음.
따라서 여러 분자의 random walk를 여러 CPU 코어에서 동시에 실행할 수 있음.

비유하자면, 2,936장의 시험지를 채점할 때:
- 기존: 채점자 1명이 한 장씩 순서대로 채점 (2,936장 순차 처리)
- 개선: 채점자 100명이 각자 약 30장씩 나눠서 동시에 채점 (병렬 처리)

`--n_jobs` 옵션으로 동시에 사용할 CPU 코어 수를 지정:

```bash
# 기존과 동일한 순차 실행 (기본값, 1명이 다 처리)
python SSM/bin/ssm_smiles.py --train_data ... --test_data ... --DiSC 3

# 100개 코어로 병렬 실행 (100명이 나눠서 처리)
python SSM/bin/ssm_smiles.py --train_data ... --test_data ... --DiSC 3 --n_jobs 100
```
